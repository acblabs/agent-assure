# Agentic Measurement Use Case

This brief frames a reproducible measurement problem for AI agent governance
pipelines: a candidate implementation can preserve the visible task answer
while regressing the evidence, policy, redaction, escalation, review, live
protocol, or observable trajectory behavior around that answer.

The fixture method is intentionally deterministic. `agent-assure` holds model
outputs, tool outputs, requests, expectations, and fixture manifests fixed,
then evaluates the candidate RunSet against labeled expectations and built-in
governance controls. Baseline comparison is secondary context. Hashes and
digests describe provenance; they are not behavioral verdicts. The live method
is intentionally protocol-bound: repeated stochastic observations are
interpreted only under declared provider/model windows, clustering assumptions,
comparison methods, retry/exclusion rules, budgets, and data boundaries.

This is a NIST-style measurement use case because it is written for test,
evaluation, verification, validation, and assurance review audiences. It is not
a NIST submission, endorsement, safety claim, compliance claim, or clinical
validation claim.

## Measurement Question

Can a deterministic fixture-mode assurance harness detect governance-pipeline
regressions that are missed by raw answer comparison or provenance hashing?

The deterministic fixture-mode question is narrower:

```text
Given a fixed fixture suite and compiled labeled expectations, does the
candidate RunSet violate any expected outcome, material evidence-link invariant,
provider/tool policy, redaction boundary, prompt-boundary rule, escalation rule,
or human-review control?
```

The live stochastic question is:

```text
Given a frozen live protocol, do repeated observations preserve declared
expectation-pass, reason-code, outcome, exclusion, cost, latency, trajectory,
and operational event-process bounds under the stated clustering and
comparability assumptions?
```

## Unit Under Test

The unit under test is the governance pipeline around an agentic task, not the
underlying language model. The repository examples exercise:

- output categories and allowed outcomes;
- fixture-declared material claims and structured evidence links;
- provider and tool allowlists;
- prompt-injection boundary cases;
- redaction checks over persisted summaries and reports;
- required human-review routing;
- runtime failure capture;
- fixture equivalence and provenance diffing.

The current implementation includes explicit live-adapter commands for
protocol-bound repeated observations and operational rate, cost, and latency
reports. These reports do not establish safety assurance, prove regulatory
compliance, validate clinical workflows, provide general provider-quality
evidence, or support general stochastic model-quality claims.

For live reports, the unit under test also includes the declared adapter and
execution boundary. The implementation provides static JSONL, no-shell
external-script, and OpenAI-compatible chat-completions adapters. The
OpenAI-compatible adapter uses Python standard-library HTTP support, requires
explicit network opt-in, and enforces HTTPS plus endpoint-host allowlisting.

## Measurement Method

1. Author a YAML suite with cases, shared fixture roots, and expectations.
2. Compile the suite into strict JSON with resolved expectations.
3. Build a fixture manifest with POSIX-normalized paths and SHA-256 file
   digests.
4. Run the baseline and candidate variants against identical local fixtures.
5. Evaluate each RunSet against expectations and deterministic controls.
6. Compare baseline and candidate only after fixture equivalence passes.
7. Publish JSON and Markdown reports, an evidence packet, and release replay
   digests for review.

The primary pass/fail decision comes from the candidate versus expectations.
The baseline helps reviewers understand whether a change is a new failure,
resolved failure, unchanged failure, or unchanged pass under equivalent
fixtures.

## Flagship Result

The flagship showcase is the prior-authorization evidence-normalization
candidate. The baseline and candidate use the same compiled suite and fixture
manifest. The baseline passes all ten fixed cases. The candidate keeps the
visible answer stable for the shared-source multi-claim case:

```text
recommendation=approve
outcome=approve
```

The candidate still fails because the fixture-declared material claim
`claim-duration` loses its structured evidence link.

```text
case_id: shared-source-multi-claim
control_id: material_claims_have_evidence
target: claim:claim-duration
reason_code: MATERIAL_CLAIM_MISSING_EVIDENCE
```

The baseline-to-candidate comparison report classifies the result as
`new_failure` with fixture-equivalence state `pass`. The generated report is
human-reviewable because it shows the failing case, control, target, reason
code, missing evidence link, fixture-equivalence result, behavioral evidence
reference change, and separate provenance changes.

## Why The Result Matters

The example is deliberately not a live model-quality benchmark. Its value is
that it isolates a governance-pipeline failure mode:

- raw answer comparison would see the same visible answer;
- prompt, configuration, and code digests would show that something changed but
  would not say whether the change was acceptable;
- the material evidence invariant identifies the missing link that matters for
  review.

This is the core contribution of the current implementation: make deterministic
governance regressions reproducible, reviewable, and distinct from provenance
changes.

## Interpretation Boundaries

The result tables are empirical only in the fixture-bound sense. They count
deterministic cases, findings, and reason codes produced by repository
artifacts. They do not estimate live model accuracy, provider reliability,
population risk, safety, regulatory compliance, or clinical validity.

Unsupported capabilities are marked `not_evaluated` rather than treated as
passing. This is a deliberate measurement choice: absence of evidence is kept
visible instead of being collapsed into a green check.
