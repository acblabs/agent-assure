# agent-assure v0.3.1

v0.3.1 adds the measured-usage schema foundation while keeping the project
inside its evidence boundary. Usage artifacts report measured token, tool-call,
retry, latency, and declared estimated cost fields for human review. They do
not claim business impact.

The active frozen schema snapshot is now `schemas/v0.3.1`. The v0.3.0 snapshot
remains unchanged for replay, and current models continue to accept legacy
`schema_version: 0.2.0` artifacts while emitting `schema_version: 0.3.1` for
new artifacts.

Usage hardening in this release includes:

- integer `estimated_cost_microusd` instead of float money fields;
- explicit limitations required on cost-bearing usage segments;
- ledger/summary consistency checks;
- partial missingness limitations carried into summaries and deltas;
- rejection of legacy-labeled containers that carry v0.3.1 usage fields;
- evidence-packet Markdown that surfaces whether usage summaries are present.

The release also tightens claim-boundary checks for return-on-investment
wording.

Post-release hardening now verifies downloaded release-bundle cosign signatures
in the PyPI publish job before digest replay or package staging, rejects
release manifest paths that escape the selected artifact root, records per-step
release build logs under the release output directory, pins release
reproducibility inputs with `SOURCE_DATE_EPOCH` and fixed `--today`, and keeps
redaction recursive for nested values under preserved structural keys.

Additional hardening keeps public claim-boundary checks on `CHANGELOG.md` and
release notes, centralizes frozen-schema packaging checks, preserves live usage
counters on post-response budget failures, keeps observed live failures
verdict-bearing when a run is incomplete, and requires claim-evidence links to
target present `evidence_items[].ref_id` values. The evidence-diff HTML renderer
uses the same link contract, source git/lockfile provenance remains rooted at
the source project when report outputs are external, and report labels now call
the recommendation/outcome comparison decision-field equivalence. The flagship
golden report is generated from the same explicit claim-link contract, the
final release workflow runs the canonical `make release-check` gate, and
TestPyPI/local release verification uses the locked editable install pattern.
